= Android codegen
:toc: macro

`codegen.zig` emits gossamer's Android JVM-bytecode shims into the marked-generated tree `generated/android/`. This is the tooling that satisfies the estate rule that any unavoidable JVM shim is *generated by gossamer's own tooling*, not authored by hand and not pushed onto downstream apps.

== Why this exists

Android's `Activity`, `Service`, `BroadcastReceiver` and `AppWidgetProvider` are JVM classes; the OS instantiates them by name and drives their lifecycles. gossamer cannot avoid having *some* JVM presence, but it keeps that presence (a) tiny, (b) generated, and (c) a pure forwarder to native handlers. Downstream apps never see Kotlin/Java — they register Rust/Zig/Idris handlers and ship `generated/android/` verbatim.

== Run

[source,sh]
----
# from the repo root
zig run tools/android-codegen/codegen.zig -- [package] [label] [launcherActivity] [widgetInfoRes]
# or via the wrapper:
just android-gen
----

The five component shims are app-agnostic and emitted verbatim (with a do-not-edit banner). Only `AndroidManifest.template.xml` is parameterised — that is the one artifact that legitimately varies per app.

== Layout

* `templates/*.java.tmpl` — canonical shim sources (gossamer tooling inputs; deliberately not `*.java`). Each native method here corresponds to an `export fn Java_io_gossamer_*` in `src/interface/ffi/src/`.
* `templates/AndroidManifest.xml.tmpl` — manifest skeleton with `{{PLACEHOLDER}}`s.
* `codegen.zig` — stamps the banner, substitutes manifest placeholders, writes `generated/android/`.

== Invariant

Editing a `.tmpl` and re-running the generator is the only supported way to change `generated/android/`. If you add or rename a native method in a template, add the matching `export fn` in the corresponding `android_*.zig` (the symbol names must stay in lock-step — JNI resolves them by the `Java_<pkg>_<Class>_<method>` mangling).
